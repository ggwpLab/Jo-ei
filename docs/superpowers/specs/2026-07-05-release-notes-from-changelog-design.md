# Release Notes from CHANGELOG.md — Design

Date: 2026-07-05
Status: approved

## Problem

The v0.1.0 GitHub release notes were auto-generated by goreleaser from the
full commit history (v0.1.0 is the first tag, so "commits since previous tag"
meant every commit in the repository). The result is hundreds of lines of
`feat(...)`/`fix(...)` subjects prefixed with full 40-character SHAs —
unreadable, and completely disconnected from the hand-curated
Keep-a-Changelog `CHANGELOG.md` that ships in the repo. Two changelogs exist
and they do not agree.

## Decision

`CHANGELOG.md` is the single source of truth for release notes. The release
workflow extracts the tagged version's section and passes it to goreleaser
via `--release-notes`. goreleaser's own changelog generation is disabled.
The existing v0.1.0 release notes are replaced retroactively with the
curated 0.1.0 section.

Rejected alternatives:

- **Improve auto-generation only** (`changelog.use: github`, PR titles):
  more readable, but the two-changelog divergence remains.
- **Hybrid** (curated section + collapsed auto commit list): extra workflow
  complexity for a commit list nobody asked for; the compare link at the
  bottom of each CHANGELOG.md section already covers "full diff" needs.

## Components

### 1. `scripts/release-notes.sh`

POSIX shell + awk script (runs on ubuntu CI and locally in Git Bash).

- **Input:** version as `$1`, without the `v` prefix (e.g. `0.2.0`), and
  optional `$2` as the path to the changelog (default `CHANGELOG.md`).
- **Output:** the body of the `## [<version>]` section (everything between
  that heading and the next `## ` heading or the link-reference block),
  trimmed of leading/trailing blank lines, followed by a blank line and:

  ```
  **Docker image:** `docker pull ghcr.io/ggwplab/jo-ei:<version>`
  ```

  The footer moves here because goreleaser ignores `release.footer` when
  `--release-notes` is supplied.
- **Failure modes (exit 1, message to stderr):**
  - no `## [<version>]` heading found;
  - the section exists but contains no content (only blank lines).
- The script must not match the `[Unreleased]` section or the
  link-reference definitions at the bottom of the file.

### 2. `.github/workflows/release.yml`

New step between checkout/setup and the goreleaser step:

```yaml
- name: Extract release notes from CHANGELOG.md
  run: ./scripts/release-notes.sh "${GITHUB_REF_NAME#v}" > /tmp/release-notes.md
```

This doubles as the guard: tagging a version whose section is missing from
`CHANGELOG.md` fails the release before any build or Docker push happens.

The goreleaser step's args become:
`release --clean --release-notes=/tmp/release-notes.md`.

### 3. `.goreleaser.yaml`

- Replace the `changelog:` block (sort/groups/filters) with
  `changelog: { disable: true }`.
- Remove the `release.footer` block (moved into the script output).

### 4. `CONTRIBUTING.md` — "Release process" section

Short checklist for maintainers:

1. Move the `## [Unreleased]` content into a new `## [X.Y.Z] - YYYY-MM-DD`
   section; leave `[Unreleased]` empty.
2. Update the link references at the bottom (`[Unreleased]` compare range,
   new `[X.Y.Z]` link).
3. Optionally dry-run `./scripts/release-notes.sh X.Y.Z` locally.
4. Commit via the normal PR flow, then tag `vX.Y.Z` on main and push the
   tag. The Release workflow does the rest.

### 5. Retroactive v0.1.0 fix

One-off command, run once after the script lands:

```
./scripts/release-notes.sh 0.1.0 > notes.md && gh release edit v0.1.0 --notes-file notes.md
```

Not part of CI; executed manually during implementation.

## Testing

Script-level tests run against the real `CHANGELOG.md` plus fixture inputs:

- `0.1.0` → outputs the full curated section + Docker footer, exit 0.
- Missing version (e.g. `9.9.9`) → exit 1, error names the version.
- Version present but empty section (fixture) → exit 1.
- Output must not contain the `## [0.1.0]` heading itself, the
  link-reference lines, or any `[Unreleased]` content.

CI-level verification: goreleaser local dry run
(`goreleaser release --snapshot --clean --skip=docker`) still passes with
the changelog block disabled.

## Out of scope

- Changing the CHANGELOG.md format or backfilling more history.
- Automating the Unreleased → versioned-section move (release-drafter,
  semantic-release, etc.). Manual curation is the point.
